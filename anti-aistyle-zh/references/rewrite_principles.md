# 改写原则

这份文件只补“怎么改”，不重复主合同。
去 AI 味优先恢复作者性、局部判断、结构误差和运行差异，不优先追求口语化，也不是为了骗过检测器；对 `3000+` 字长文，目标是功能保真，不要把正文做成摘要化改写。

## 快速路由

- 默认改写动作先查第 1-2 节。
- 评论、方法文查第 3 节；叙事、非虚构查第 4 节。
- 小说章节查第 5 节；提纲和剧情规划查第 6 节。
- 用户显式要求局部替代时查第 7 节。
- 事实、体裁、检测器化和越界底线查第 8 节。
- 本文件讲改法，不替代主合同和专项 reference。

## 1. 六个基本动作

按这个顺序改：

1. `删`
   - 二次解释
   - 二次总结
   - 二次证明
   - 连续解释链里最后那层意义翻译
   - 结尾多补的一刀
2. `压`
   - 三个同功能例子压成一个
   - 三层说明压成一层
   - 连续同功能段压到两段以内
   - 但不要压成更短、更稳、更容易复述的标准答案
3. `并`
   - 把必要信息并回动作、景物、事实、结果
   - 不让它单独站成“答案句”
4. `换`
   - 总结句换具体后果
   - 抽象判断换动作
   - 机制名词串换具体场景或具体后果
   - 工整结尾换自然停住
5. `补`
   - 先重排原文已有物件、动作、对话、地方介质
   - 小说 / 叙事可再补原文已暗示、但没展开的介质
   - 事实类文本只允许重排原文已有信息，或把抽象判断落回原文已经出现的事实
   - 新增叙事细节只有在能改变段落功能、scene 运行或结构重心时才算有效
   - 原文没有支撑的“更具体”不是去味，是越界
6. `收`
   - 事实和情绪已经成立时，及时停住

## 2. 改写优先级

第一优先级：

- 作者替读者解释
- 连续解释链
- 工整对比句替读者归纳意义
- 小说里作者半句翻译
- 段尾总结 / 盖章
- 汇报腔、课件腔、安全表达
- 连续段功能重复
- 长距离重复论证
- 段首领句 / 报题
- 成熟评论员人格 / 成熟回望人格
- 情绪落锤模板簇
- 动作后的心理翻译
- 情节功能过顺

第二优先级：

- 重复举例
- 重复证明
- 重复抒情
- 步骤骨架过强
- 模板化推进
- 反方限定过密

第三优先级：

- 抽象词替换
- 机制名词串降密
- 黑话降密
- 局部句法打散

## 3. 文章 / 评论 / 方法文

先查这些骨架：

- `定义问题 -> 分层拆解 -> 价值回收`
- `判断 -> 解释 -> 例子 -> 收束`
- 每段都像要交付一个结论
- 连续 `400-600` 字不落回生活表面
- 换了几个具体名词，段落功能却没变

执行卡：

- 删一部分骨架词和软转场
- 把一部分大判断改回局部观察
- 至少拆掉一处完整 `判断 -> 解释 -> 例子 -> 收束`
- 全文超过 `3000` 字时，至少拆 `2` 处
- 允许个别段落只停在事实或经验，不急着回收到观点
- 如果原文已经在按“多层面 / 多原因 / 多维度”推进，改后不要保留等权分层梯子；至少压塌一层，不让正文继续按“一层面一段、一原因一段”齐步往前走
- 中段至少留一块只围着一个对象、一个摩擦或一处具体后果转，不要每一块都升回宏观解释
- 删解释后，不要回补更多显性转场词、安全限定句、填缝副词或否定-肯定骨架
- 如果有连续 `150-300` 字都在替同一个对象做价值阐释、用途申辩或意义回收，把它当成一个热区整块拆，不要逐句抛光
- 遇到“生活经验 -> 公共论述 -> 高位结论”一口气铺满的长段，优先打断成对象、动作、阻力、后果，不要只把句子修顺
- 如果原文只有一处解释热区，改后却拆成多处更短、更匀、更像成稿卡片的解释区，也算失败；不要把一段解释拆成三段清楚话
- 如果一个解释死结被拆成两三处更顺的 `判断句群 / 例子群 / 价值回收群`，哪怕分散到了别段，也仍算同一种失败；不要把一团硬解释改写成多块平顺解释
- 评论总论如果长成 `别把X当Y / 真正该先问的是 / 问题不在A而在B` 这种稳起手，再接 `表面问题 -> 真问题 -> 方法 -> 结语` 的成熟专栏骨架，默认继续拆
- 如果改后只剩更短、更稳的提纯稿，说明压过头了
- 把一部分“问题 / 状态 / 机制”改回人、动作、对象和后果
- 至少留一处判断仍带认识过程，不要句句都像结案意见
- 如果原文明显在写给读者看，不要把它修成说给人听的口播稿
- 如果评论包里的总论稿一开头就稳稳给出 `这件事真正该怎么看`、后面再按 `表面问题 -> 真问题 -> 方法 -> 结语` 展开，哪怕更自然也算回退；总论稿也要拆掉一处成熟评论壳
- 评论包里的补充评论稿不要写成“前两份已经说了什么，这里再把结论收回来”的收官稿；补充稿应补偏、补窄、补一处卡点，不重新总括整包
- 评论包里的中段稿、补充稿不要起手就把对象缩成一个“边角料小设施 / 小切口 / 小问题”，再顺手升成成熟专栏；如果一开头先写“它不大 / 看起来不复杂 / 好像只是……”，后面又接“真正稀缺的是 / 关键是它能不能 / 更重要的是”，默认继续拆
- 具体设施评论不要靠一串 `有人……有人……也有人……` 把对象用途论证圆；这类例子串常会把正文推回“用途申辩稿”，至少拆断一处，不让它把价值一路讲稳
- 如果一篇评论已经是组稿中的第二篇、第三篇，不要再写成“先替对象正名 -> 再解释为什么不止如此 -> 再落公共价值”的完整小专栏；组稿内单篇允许只卡在一个摩擦点上
- 不要把尾段重新修成 `综合判断 -> 导向句 -> 高位结论块`
- 如果尾段仍像成熟评论稿在稳稳落版，继续拆；宁可收在未讲圆的摩擦、具体代价或对象压力上，也不要盖一个太完整的大结论
- 政策评论如果越改越像成熟评论员定稿，优先拆最后两段的宏观结论，换回地区、家庭、执行摩擦或判断分叉
- 政策评论混合体不要追求“每层都讲到、每段都讲满、首尾都完整”；少一层、偏一边、留一处没讲透，往往比一篇平衡成稿更安全
- 如果原文是“政策评论包 / 评论组稿 / 方法评论混合件”中的一篇，不要把它独立修成一篇闭环专栏；允许它保留一点包内文稿的非对称和未盖棺感

如果改后明显变短，要能指出删掉的是哪些重复功能、护栏段、双面铺垫或空转解释。

频率提醒：

- 不要为了把句子缝顺，回补 `其实 / 已经 / 仍然 / 于是 / 同时 / 进一步` 这类填缝词
- 也不要把句子普遍改成 `不是……而是…… / 不只是……更…… / 不是没有……只是……` 这类稳定骨架
- 检查的不是单个词或单个句式能不能出现，而是它们在段首、转句位、收束位是否连续抬头
- 频率判断优先和原文比，再和同体裁正常成稿的存在感比；不是越少越好，而是不要高过该体裁真人常态

篇章与节奏提醒：

- 不要只换词；要看相邻段是否仍在复写同一个判断
- 至少拆掉一处过稳的段群关系，不让段与段总按同一种因果 / 转折 / 回收关系接力
- 不要只把句子修得更顺；要同时看长短句、断句、标点是不是仍然同色
- 标点不是装饰；它应该服务现场、对象和节奏，而不是只维持整洁

正式壳提醒：

- 不要把正文越改越像非人称说明文；要看句子是不是在往名词化、去主体、去动作的方向收缩
- 能写成“谁在做 / 谁在承受 / 谁在判断”的，尽量不要写成抽象状态自己发生
- 不要把本来有人、有对象、有摩擦的段落洗成“问题 / 条件 / 约束 / 状态”自行展开的成稿壳
- 不要为了稳妥，把认识过程全改成已经定稿的结论块
- 如果一段正式说明越改越像“成熟制度释义稿 / 标准公文说明块”，优先删掉那层替对象申辩的价值解释，只留动作、条件、范围、后果和执行关系
- 如果出现整段 `问题导向 / 目标导向 / 过程导向 / 结果导向`、`统筹 / 协同 / 压实 / 闭环` 这类总括政策句群，把它当成解释热区整块拆；不要只是换词保留同一段大而平的政策说明

编号执行摘录 / 条文说明：

- 原文已经是编号摘录、条文口径说明、执行问答或操作摘要时，优先做 `低介入去壳`，不要重写成更完整、更体面、更像制度释义稿的成稿
- 优先保留原条文顺序、编号块感和执行口径，不要把它铺成一篇连贯大文章
- 小标题能直接落主题的，不要机械重复 `关于`
- 开头若是 `本文供……使用 / 不能替代……但可用于…… / 现作如下说明` 这类主持块，优先压到最短；不要保留一整段前置说明壳
- 不要为了让句子更顺，回补 `并非 / 并不 / 不是……而是…… / 不只是……更…… / 仍 / 已经` 这类整理词和否定骨架
- 能直接写动作、条件、范围和后果的，就不要再加一层“解释为什么不是那个意思”的主持句
- 原文已经正式时，只减空转解释、空转礼貌和空转限定；不要把它抬成更法言法语、更像示范文本的版本
- 如果改后 `关于 / 为进一步 / 现就 / 仍应 / 不应简单理解为` 这类主持虚词和否定解释链抬头，哪怕句子更顺，也要继续压
- 如果改后开始像培训手册、统一口径解读稿、十一条完整填报说明，而不是原始的执行摘录 / 简明说明 / 现场口径，也算回退；正式文优先保“够用”，不要改成“讲透”
- 表格填写说明、执行口径摘录、操作摘要这类文本，优先保短条、问答、注意项、误区项和字段边界；不要顺手扩成 `适用范围 -> 记录时段 -> 计数单位 -> 结语` 的完整解读件
- 知识库、客服口径、FAQ、内部周报、使用说明这类结构化材料，`低介入` 不是把每句话都拆成独立短条；如果改后关键词复用更密、句长更齐、段落全变成一行一项，就合并一部分相邻动作 / 条件 / 例外，保留少量不等长说明和必要上下文
- 评审意见回复、修改说明、审稿回复这类文本，保留逐条回应格式，但不要把每条都压成同一种 `已补充 / 已调整 / 本研究不用于... / 后续可...` 模板；能落到具体修改位置、材料边界或不改原因的，直接写，不必每条都完整铺一遍限制、依据和展望

口播 / 社媒脚本：

- 原文已经是口播稿时，去味不是把它改成使用手册或 FAQ；不要连续堆 `怎么借 / 怎么还 / 注意事项 / 联系方式` 这类卡片题头
- 保留口播的连贯说话气和现场对象，操作信息可以嵌回场景句、提醒句和一句带过的动作里，不要把每个动作都拆成等长指令
- 如果改后主要由短标题 + 命令句 + 禁止句组成，虽然更清楚，也算口播体裁被洗成说明卡

## 4. 叙事 / 随笔 / 非虚构

先查这些路线：

- `现场 -> 回望 -> 领悟`
- 返乡 / 回忆 / 旧地重访走得太标准
- 复杂感受总被讲圆
- 人物总被写成某个议题的承载器

执行卡：

- 内部先形成一句主导路线，默认不外显
- 让感受停在动作、景物、停顿和局部话语里
- 少替情绪命名，少替人物补明确动机
- 至少拆断一处 `现场 -> 回望 -> 领悟`
- 长文超过 `3000` 字时，至少拆 `2` 处
- 补细节优先挪用原文已有物件、动作和对话
- 允许局部卡一下、空一下，不要每一步都正好服务下一步
- 有引语或对话时，先保对象感、身份差和临场感，不要先修成成熟说明句
- 至少留一个可进入的人物视角支点：代词落点、时空定位、目光落点、身体阻力、注意力偏移、误听 / 迟疑 / 未接住的动作
- 不要把人物经验只改成“信息更清楚”；要让读者还能进去
- 原文带地域、阶层、社群、职业或代际声纹时，不要顺手洗成统一标准散文

人物特写 / 纪实非虚构：

- 删掉一部分“借人物整理答案”的高位句
- 让更多段落停在工作动作、房间状态、家属反应和未说完的话
- 不要稳稳收在“他守住的不只是生计，也是体面 / 活路 / 手艺”这类人物定稿尾段
- 如果改后形成 `人物肖像段 -> 一件事 -> 现实意义 / 公共价值回收段` 的新外壳，继续拆；不要把去味稿写成更成熟的人物报道成稿
- 中长人物稿也不要滑成 `铺面介绍 / 动作观察 / 旁观者认可 / 人物价值落稳` 的完整敬服稿；一旦读起来太“可信、可靠、让人服气”，先拆那层认可回收

短人物稿：

- 不要默认写成“出场肖像 -> 一件小事 -> 手艺或品性落稳”的标准人物成稿
- 少用一开头就很完整的外形 / 年龄 / 从业年数 / 人生态度打包介绍
- 至少留一处不为人物定性服务的小摩擦、没接住的话或工作里的别扭动作
- 如果整篇读下来太像报刊副刊人物小稿，继续拆
- 第一段优先从摊位、手上动作、顾客一句话或当天的一件小麻烦切入，不要先把人物“介绍完整”
- 结尾优先停在收摊、回家、下一单、手上余动作，不要总落成人物价值盖章
- 不要把人物稿改成“旁观者稳稳服气”的敬服稿；像 `这样的人 / 让人服气 / 把东西交到他手里就安稳` 这类认可回收段一旦成块出现，继续拆
- 手上动作先是动作，不急着翻成品性判断；也不要顺手把人物来路、改行原因、从业史补成完整背景段
- 对服务摊点、小店、维修、跑腿、窗口劳动者，尤其要防 `第一位顾客小插曲 -> 摊位与工具展开 -> 人流验证 -> 行业来路补全 -> 收摊回声` 的完整副刊人物壳；命中三段以上时，不要局部修，要拆整套流程
- 这类短人物稿优先保“一个横截面”，不要跨完一个完整工作日；`开摊 -> 早高峰 -> 午饭 / 午后 -> 傍晚收摊` 这种整日骨架默认算硬残留，至少砍掉其中两段
- 不要把价目、工具盒、摊位摆设、全天人流、改行原因一次补齐，也不要让开头那位顾客或结尾那次回访专门替人物立住可信度；一旦出现“放学又回来 / 傍晚再出现 / 第二次验收”这类回返闭环，优先拆
- 更硬的触发条件：短人物稿一旦同时出现 `来路回填 + 全天时段切换 + 价目/工具完整交代 + 尾段价值回声`，默认判为未完成，不要只微调句子；至少删掉其中两项
- 审计时可按 `开场事件 / 来路背景 / 工作方法 / 人流验证 / 价目规则 / 午后或傍晚余波 / 作者收束` 七格去看；填满五格以上通常已经过满，去味时要主动挖空，不要把每格都保住
- 尾段如果重新起了高位主持框，例如“如果一定要说这摊子特别在哪里 / 说到底 / 到了最后 / 留住的是……”，通常说明人物稿又回到了评论式收束；优先删掉这层主持框

## 5. 小说章节 / 长篇

先查这些残留：

- 对比句是在做辨异，还是在替作者归纳
- scene 里有没有连续 `3` 句都很稳
- 人物状态已经成立后，作者有没有再补半句翻译
- 情绪落锤模板是不是跨章节重复
- 线索、证人、物件、反转是不是按叙事功能准时排队
- 删掉显性解释后，是否换成更软的抒情翻译句

执行卡：

- 命中 `情节功能过顺` 时，先列功能链
- 不要机械删对比句
- 优先拆作者补的半句解释
- 跨章节重复模板先改一半以上
- 情绪尽量落回动作、停顿、呼吸和物件
- `平衡` 档允许极小结构调整：延迟一个证人、让一个线索先不可用、让一个物件只给半条信息
- 有对白时，不要把不同角色修成同一种成熟句法；先保人物差、场面压力和没说完的话
- 先保 scene 的进入点：谁先注意到什么、谁误判了什么、谁身体先变；必要时把代词、时空和感知锚点补回去
- 如果 scene 只剩信息和动作，还要补一个 viewpoint anchor，而不是补解释句
- 方言、俗语、行话、代际说法如果本来承担人物区分，不要为了顺滑把它们抹平
- 多人任务 scene 不要写成 `到场 -> 按名单分工 -> 取到线索 -> 赶往下一站` 的顺滑程序段；如果谁负责什么、谁留守哪里、下一步去哪里都排得太整，优先拆一处编排感
- 线索、权限单、记录簿、便签、节目单这类物件不要摆得过正、像专门等主角来取；一旦读起来像“有人替后手把理解路径铺好”，哪怕句子更自然，也算残留
- 不要把线索包写成“可直接执行的提示包”：精确到分钟的时间戳、完整的人名、稳定外貌特征、明确路线、准确柜号 / 房号 / 转角说明如果连续出现，哪怕世界观允许，也会把 scene 推回人工设计感
- 这类提示包不要层层都有效：拿到一张纸，不等于立刻知道下一个地点、下一个人、下一个动作；至少让一环只给半条信息、给错重心或暂时不可用
- 如果 scene 已经靠环境、身体、误判和关系压力立住，不要再补一套“看什么 / 去哪儿 / 找谁 / 几点前到”的便利说明；这不是清楚，是把推进轨道露出来
- 角色或旁白就算自己说了“这线索太完整 / 太像安排好的 / 太精确”，也不算自动去味；如果后续仍照着这套提示顺利推进，这种自我提醒只是把人工设计感说出来，不是把它解决掉

短 scene 再加一条：

- 不要做成“更短、更亮、更干净”的微型成稿
- 不要只剩 `几句对话 + 一个动作 + 一声响 / 一件物`
- 至少留一点不承担情绪收束的现场杂质或俗务动作

## 6. 提纲 / 剧情规划

先查这些壳：

- 每卷功能过匀
- 角色主场排班过匀
- 升级卷卷准点抬高
- 中段重复同一种揭示机制
- 地方只在服务命题
- 缺少白跑、误判、低功率小场和无功能场

执行卡：

- 拆掉一卷的标准功能
- 拖延一个角色的主场
- 让至少一卷只下挖，不升级
- 连续两次同类揭示后，第三次换发亮方式
- 补地方反向塑形剧情的段落
- 补白跑、误判、俗务、缓冲和后效
- 如果是单卷或中短规划，去壳后仍要像规划文本，不要顺手改成一篇流畅卷介绍
- 单卷、章节群或卷群规划，优先保留“块感”和工作感；允许短块、断裂句、局部列点和密度不齐，不要把整份规划铺成连续大散文
- 每一块先写成事点、阻力点、变数和回收点，不要为了自然把同一块扩成完整论证段
- 保留阶段阻力、任务错位、节奏不齐和功能差，不要把每段都写成顺滑 prose
- 如果改后更像可直接刊发的卷介绍、项目导览或大段摘要，而不是案头规划文本，默认继续回写
- 如果去壳后整份文本开始像 `项目说明书 / 卷介绍 prose / 方案说明文`，即使语气更自然也算回退；规划文本宁可留工作块和未抹平的接口，也不要被抹成一篇完整说明文

## 7. 局部替代改写

如果用户切到 `局部替代改写`，最小约束如下：

- `更稳`：不新增动作、背景、心理、关系，只做减法和替换
- `更自然`：可补一处原句已暗示的介质，但不新造背景、动机、关系
- `更贴近样文`：贴节奏，不贴现成句，不复制样文救火句法

一个局部默认只改：

- `1` 个主句
- `1` 个收束动作
- `1` 个辅动作

不要一口气把姿态、结构、口吻、信息密度全部换掉。

如果局部里同时有引语、对话或对象关系：

- 优先保说话人的位置和口气
- 不要顺手补一层主持人口吻把话讲圆
- 不要把原本有棱角的话改成可直接引用的成熟句子
- 不要顺手把地方话、行话、代际词法统一成标准语

## 8. 底线

不要做：

- 偷换叙述视角
- 把评论硬改成戏剧现场
- 把小说现场抬成高位总结
- 为了去味把保留型对比句一刀切掉
- 为了更像人而故意加病句
- 为了降低 AI 感故意制造错别字、口头禅、情绪噪音、无意义停顿或碎片化
- 为事实类文本补原文没有的事实、数据、身份、场景、时间地点、因果链或人物动机
- 擅自新增关键设定、改名词体系、改立场

强体裁 / 设计外：

- 正式讲话、政策评论、执行口径文，不要只保留体裁外壳，也不要只保住成熟成稿壳
- 法规说明、办法摘录、执行问答类文本，允许保留编号、边界和责任链，但要继续拆过密的解释壳、限定壳和主持人口气
- 如果文本本质上是给执行者看的说明文，允许合并同类条目，改按风险或操作顺序组织，不必把每一条都扩写得一样完整
- 对法规说明、执行摘录、通知问答类文本，不要把原本已经能执行的条目再翻成“为什么这样规定 / 风险在哪里 / 价值是什么”的主持说明；能直接落回触发条件、动作、时点、责任和回执的，就不要再补一层讲解
- 工作通知、材料发放说明、问答摘录类文本，不要顺手写成“抬头称呼 + 顺利推进背景 + 现就如下说明”的完整通知件；能直入使用边界、任务差异和操作条件的，直接进
- 不要让前言长成一整段发布会式总说明；元说明只保留必要部分，其余让位给实际使用信息
- 材料发放通知、使用说明、问答包，不要一份文件里同时长成“背景导入 + 元说明 + 使用手册 + FAQ 教程 + 安抚总结”的完整培训稿；先保收件对象、阅读顺序、提交动作、反馈口径、保密边界和时间安排，其他能短则短
- 讲话、访谈、纪实、公文里有对象时，先保对象、关系、用途和现实压力，不要先洗成通用自然散文
- 短正式讲话若还完整保留 `开场问候 -> 借机讲三点 -> 价值收束 -> 感谢`，默认继续回写
- 法规说明 / 执行口径文不要只做顺滑化；如果只是更像规范说明件，不算完成
- 原文已经很正式时，不要再往更法言法语、更书面同义词、更成套四字稳句上抬；这类“升级正式度”通常只是在加 AI 壳
- 不要把原本已经够用的执行摘录，重写成更像制度解读、发布说明或示范文本的版本
